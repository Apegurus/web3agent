### Findings

#### 🟠 Implicit CCXT order endpoints still bypass financial spend-limit enforcement

- **Category:** correctness
- **File:** `src/ccxt/classification.ts:67`
- **Severity:** high | **Confidence:** high | **Status:** validated
- **Claim:** CCXT implicit order-placement methods such as `privatePostOrder` or `fapiPrivatePostOrder` are classified as `private_write` but not as financial writes, so they skip USD-based policy enforcement.
- **Evidence:** `CCXT_FINANCIAL_WRITE` only includes `createOrder`/`editOrder`; implicit private methods are allowed; `classifyCcxtWriteRisk()` defaults non-listed private writes to `destructive`; `ccxt_private_write` uses that classifier for runtime and queued risk.
- **Why it matters:** An agent can place exchange orders through implicit endpoints while avoiding financial spend-cap checks.
- **Suggested fix:** Treat implicit order-creation/edit endpoints as financial, or conservatively deny private-write methods whose names indicate order placement when USD estimation is unavailable.
- **Consensus:** Flagged by generalist OpenAI.

#### 🟠 Policy-enabled CCXT create/edit orders are still blocked for non-USD quote pairs

- **Category:** correctness
- **File:** `src/policy/extract-usd.ts:39`
- **Severity:** high | **Confidence:** high | **Status:** validated
- **Claim:** CCXT `createOrder`/`editOrder` estimation returns `0` for non-USD quote pairs like `ETH/BTC`, causing policy denial as `USD_ESTIMATION_FAILED`.
- **Evidence:** `extractCcxtOrderUsd()` returns `0` unless the quote asset is in `USD_QUOTES`; `USD_QUOTES` only includes USD/stablecoin variants; `evaluatePolicy()` denies `estimatedUsd === 0`.
- **Why it matters:** Legitimate non-USD-denominated CCXT orders remain unusable whenever policy enforcement is enabled.
- **Suggested fix:** Add quote-asset USD conversion or explicitly document/reclassify unsupported non-USD quote orders.
- **Consensus:** Flagged by generalist Kimi and GLM.

#### 🟡 Queued CCXT private writes are not durable across restart

- **Category:** correctness
- **File:** `src/tools/ccxt/index.ts:280`
- **Severity:** medium | **Confidence:** high | **Status:** validated
- **Claim:** `ccxt_private_write` persists queue entries that cannot be restored after restart because execution depends on an in-memory closure and no reload executor is registered.
- **Evidence:** The handler comment says restarted processes drop persisted `ccxt_private_write` entries; `loadQueue()` skips persisted operations without executors; tests assert such entries are rewritten away.
- **Why it matters:** A user can receive a pending confirmation ID, restart the process, and silently lose the operation.
- **Suggested fix:** Make queued CCXT writes restorable, or mark them explicitly non-durable/in-memory.
- **Consensus:** Flagged by generalist GLM and AI-slop specialist.

#### 🟡 CI no longer validates Node 22 and never blocks on audit findings

- **Category:** release-process
- **File:** `.github/workflows/ci.yml:22`
- **Severity:** medium | **Confidence:** high | **Status:** validated
- **Claim:** CI runs Node 24.x only while packages advertise `node >=22`, and `pnpm audit --prod` is non-blocking.
- **Evidence:** Workflow matrix only includes `24.x`; root and create package engines remain `>=22`; audit command uses `|| true`.
- **Why it matters:** Supported-floor regressions and dependency advisories can ship without the main CI gate catching them.
- **Suggested fix:** Add Node 22.x back or raise the engine floor; define which audit severities fail CI.
- **Consensus:** Flagged by generalist Claude and security.

#### 🟡 Tools subcommands emit JSON validation errors without `--json`

- **Category:** api-contract
- **File:** `src/cli/commands/tools.ts:75`
- **Severity:** medium | **Confidence:** high | **Status:** validated
- **Claim:** `tools describe`/`tools call` validation errors go through `failJson` regardless of JSON mode, so non-JSON invocations emit JSON envelopes to stdout.
- **Evidence:** `runToolsCommand()` computes `isJsonMode` but calls `failJson` unconditionally for several validation failures; `cli.ts` serializes `CliExitError` as JSON.
- **Why it matters:** Interactive CLI behavior diverges from the expected non-JSON stderr/help UX.
- **Suggested fix:** Only throw JSON envelopes in JSON mode; otherwise print human-readable errors.
- **Consensus:** Flagged by generalist OpenAI.

#### 🟡 Host-writer malformed-config protection misses valid JSON with wrong shape

- **Category:** correctness
- **File:** `src/hosts/writers/base.ts:112`
- **Severity:** medium | **Confidence:** high | **Status:** validated
- **Claim:** `readJsonFile()` rejects parse errors but still accepts arrays/strings cast to `Record<string, unknown>`, so valid-but-malformed configs can still be overwritten.
- **Evidence:** `readJsonFile()` returns `JSON.parse(raw) as Record<string, unknown>` without top-level object validation; writer code spreads/updates the result; tests cover invalid syntax only.
- **Why it matters:** Configs like `[]` or `"oops"` can still be rewritten despite the malformed-config safety intent.
- **Suggested fix:** Require parsed JSON to be a plain object or throw `HOST_CONFIG_MALFORMED`.
- **Consensus:** Flagged by AI-slop specialist.

#### 🔵 Confirmation remains a speed bump, not an authorization boundary

- **Category:** auth/authorization
- **File:** `src/wallet/confirmation.ts:37`
- **Severity:** low | **Confidence:** high | **Status:** validated
- **Claim:** The same session can enqueue and confirm pending operations, including disabling future confirmation after one self-confirmed operation.
- **Evidence:** Confirmation queue docs state it is not a caller-identity boundary; `confirm()` accepts any caller with the ID; disabling confirmation routes through one queued destructive write.
- **Why it matters:** This is weaker than a true human-in-the-loop authorization model.
- **Suggested fix:** Add out-of-band/session-separated confirmation for high-risk actions if true authorization is required.
- **Consensus:** Flagged by security.

#### 🔵 CCXT config lifecycle fails poorly for missing credentials and key rotation

- **Category:** correctness
- **File:** `src/ccxt/factory.ts:103`
- **Severity:** low | **Confidence:** high | **Status:** validated
- **Claim:** Private CCXT invokes do not preflight missing credentials, and cached runtime state assumes config file contents are immutable for the process lifetime.
- **Evidence:** `getPrivateExchange()` checks account existence but not credentials; `accountHasCredentials()` exists; private invoke paths call the factory directly; runtime-state comments document stale cache behavior.
- **Suggested fix:** Preflight credentials on private invoke paths and document or implement config invalidation.
- **Consensus:** Flagged by generalist Claude.
